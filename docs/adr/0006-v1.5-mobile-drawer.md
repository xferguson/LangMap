# ADR 0006: v1.5 Mobile drawer + touch-target sizing

## Status: Accepted (amends 0001, 0002, 0003, 0004, 0005)
## Amendment 1 (2026-05-02): Decision §6's safe-area-inset wording was wrong

The original Decision §6 line "Add `padding-bottom: env(safe-area-inset-bottom, 0)` on `.side-panel`" misidentified the target element. The Leaflet attribution control renders inside `.leaflet-container`, not `.side-panel` (which is the drawer, translated off-screen by default on mobile). On notched devices the attribution was overlapped by the home indicator. Corrected: the safe-area-inset-bottom padding goes on `.leaflet-container` (or equivalently `.leaflet-control-attribution`) inside the mobile media query. QA caught this during manual verify; new regression test at `tests/e2e/v1.5.spec.ts:153` (`v1.5-8 — Leaflet attribution clears the home-indicator safe area`).


## Context

v1.4 shipped (ADR 0005): mobile-unfriendly. The grid `360px 1fr` in
`src/styles.css:21` makes the side panel consume 84–96% of a typical
phone viewport (375–430 px), leaving the map crushed. PRD v1.2-2 /
v1.4 ACs were always written for desktop. PRD v1.5 (criteria v1.5-1 …
v1.5-9) defines the mobile experience: at ≤ 768 px, side panel becomes
a slide-in drawer toggled by a hamburger button; at ≥ 769 px, byte-for-byte
desktop unchanged. Touch targets bump up at every viewport. The user-approved
plan at `~/.claude/plans/my-language-map-is-polished-hollerith.md` is
the source of design intent for this ADR.

## Decision

1. **Drawer state owner.** `App.tsx` owns `useState<boolean>("drawerOpen")`.
   Default closed on every mount (not persisted — drawer state intentionally
   transient so the user always sees the map first on load). Toggle button
   sits in the map area, *outside* `<aside>`, so the state must live a level
   above both. Pass `onClose={() => setDrawerOpen(false)}` to a small
   `<DrawerHeader>` wrapper inside `<aside>` for the close button.

2. **CSS strategy: `display` flip, not unmount.** The drawer toggle button,
   close button, and overlay are always in the DOM at every viewport. CSS
   `@media (max-width: 768px)` flips visibility:
   - Toggle + close + overlay: `display: none` at ≥ 769 px, `display: flex`
     (or block) at ≤ 768 px.
   - `<aside>`: at ≤ 768 px, `position: fixed`, `transform: translateX(-100%)`
     when closed, `transform: translateX(0)` when open. Backdrop overlay
     covers the map.

   Why flip vs. unmount: jsdom (the unit test env) has no media-query
   support without a heavy polyfill. Tests can `getByTestId("drawer-toggle")`
   and click it regardless of viewport — the button is always there.

3. **Touch-target sizing applies at every viewport.** Slider thumbs
   18 px → 24 px (`src/styles.css` `.rc-slider-handle`); language-row remove
   buttons 24 px → 32 px (`.language-row-remove`); selects + Add button
   `min-height` 28 px → 36 px (`.language-picker-row select|button`). This is
   a deliberate desktop change too — modern desktop touch laptops also
   benefit, and v1.5-7 specifies "every viewport".

4. **`100vh` → `100dvh` migration.** Six selectors get the swap:
   `html, body, #root` (line 12), `.app` (line 22), `.app >
   [data-testid="world-map"]` (line 29), `.leaflet-container` (line 36),
   `.side-panel` (line 46), and the drawer's fixed-position rule when on
   mobile. Use `100dvh` directly (Vite + autoprefixer handle the fallback;
   if testing finds older Safari issues we add a `@supports not (height: 100dvh)`
   block as a v1.6 follow-up).

5. **Z-index layering.** Leaflet's panes top out around 700. Set:
   `.drawer-overlay { z-index: 1000 }`, `.side-panel { z-index: 1010 }` (when
   in fixed/drawer mode), `.drawer-toggle { z-index: 1020 }` (must remain
   tappable above closed drawer). Desktop layout stacking is unchanged.

6. **Focus trap — minimal viable.** When the drawer opens on mobile, focus
   moves to the close button (`<DrawerHeader>` exposes `tabIndex={0}`). When
   the drawer closes (via overlay tap, close button, or Escape key), focus
   returns to the toggle button. Full ARIA dialog (role="dialog",
   aria-modal, complete tab trap, return-focus on outside-click) is a v1.6
   follow-up if QA flags it during manual verify.

7. **Animation timing.** Drawer slide: `transition: transform 240ms
   cubic-bezier(0.4, 0, 0.2, 1)`. Overlay fade: `transition: opacity 200ms
   linear`. Both well under PRD's 300 ms ceiling (v1.5-3).

8. **`<DrawerToggle>` component.** New file `src/components/DrawerToggle.tsx`,
   props `{ open: boolean; onClick: () => void }`. Renders a `<button>` with:
   - `aria-label`: "Open menu" when `!open`, "Close menu" when `open`
     (though in practice on mobile the toggle is only visible when the
     drawer is closed; clicking it opens — but the aria-label flip future-
     proofs us if we later show the toggle while open).
   - `aria-expanded={open}`.
   - `data-testid="drawer-toggle"`.
   - Visible glyph: `☰` (U+2630).
   Minimal styling — see §10 for sizing.

9. **`<DrawerHeader>` close button.** A small new region inside `<aside>` at
   the top, visible only at ≤ 768 px (`display: none` on desktop). Contains a
   `<button data-testid="drawer-close" aria-label="Close menu">×</button>`.
   The four existing `panel-section-*` `<section>`s stay in place — header
   is rendered as a sibling above them.

10. **Toggle / close visual sizing.** Both buttons are 40 × 40 px (above the
    44-px iOS guideline and 48-px Material guideline is a v1.6 polish step;
    40 fits the existing visual weight without dominating). Toggle position:
    `top: 12px; left: 12px` inside the map area, with `padding-top:
    env(safe-area-inset-top, 0)` so it clears notches.

11. **Tablet landscape boundary.** ≥ 769 px is desktop. iPad portrait
    (768 px) gets mobile drawer; iPad landscape (1024 px) gets desktop. This
    matches the user's stated preference and avoids responsive thrash on
    rotation.

12. **Test viewport defaults.** jsdom uses ~1024 (the v1.2 polyfill in
    `tests/setup.ts` stubs 1024 × 768 dimensions). Therefore unit tests
    naturally exercise the desktop layout. The drawer + toggle + overlay are
    in the DOM and clickable; CSS `display: none` doesn't prevent
    `userEvent.click` in jsdom. Mobile-only behavior (transform animation,
    overlay-blocking-the-map at small viewports) is exercised by Playwright
    via `setViewportSize({ width: 390, height: 844 })`.

13. **`coverageMap` / `Language` shape / `langmap.v1` localStorage:
    unchanged.** Backwards-compat boundary holds.

  constraint that the existing test stays green).

## Follow-ups (recorded for the final architectural check)

- Full ARIA dialog focus management: `role="dialog"`, `aria-modal="true"`,
  complete tab trap (Tab on last focusable element returns to first),
  return-focus to the toggle button on outside-tap. v1.5 ships a minimal
  trap (focus close button on open; restore on close); remaining ARIA work
  is v1.6.
- Bottom-sheet pattern: explicitly chosen against in user-approved plan;
  if mobile UX testing reveals the drawer feels heavier than a swipe
  sheet, revisit.
- Persisting drawer-open state across reloads: deliberately not done
  (the user lands looking at the map, not the controls).
- Touch-target bump to 44×44 (iOS guideline) or 48×48 (Material): v1.5
  ships at 24/32/36/40 — adequate but not best-in-class. Heavier bump is
  a v1.6+ polish task if real-device testing flags it.
- Older Safari `100dvh` fallback: if testing on iOS 14 or earlier surfaces
  layout breakage, add a `@supports not (height: 100dvh)` block.
- Reorienting language list to a horizontal chip strip on mobile (the
  vertical list takes a lot of drawer real estate): noted as a future polish
  cycle.
