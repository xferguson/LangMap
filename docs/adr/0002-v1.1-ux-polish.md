# ADR 0002: v1.1 UX Polish + West Africa Bugfix

## Status: Accepted (amends 0001)

## Context

v1 shipped (ADR 0001) and all 16 AC pass. v1.1 adds eight new criteria
(`docs/PRD.md` v1.1-1 … v1.1-8): kill the spurious "West Africa triangles",
collapse all controls into a single side panel, replace the searchable picker
with a dropdown row, and lighten the base map (white country fill, blue
highlights). v1's data model, opacity scale, stats math, and persistence shape
are unchanged.

The triangle bug has a confirmed root cause. `world-atlas/countries-110m.json`
contains 177 country features. The inline `M49_TO_ISO` table in
`src/components/WorldMap.tsx` (165 entries) is missing **8** of them:

| `id`        | name                       |
| ----------- | -------------------------- |
| `"732"`     | Western Sahara             |
| `"238"`     | Falkland Islands           |
| `"260"`     | Fr. Southern Antarctic     |
| `"031"`     | Azerbaijan                 |
| `"010"`     | Antarctica                 |
| `undefined` | N. Cyprus                  |
| `undefined` | Somaliland                 |
| `undefined` | Kosovo                     |

`regionRefForCountry` returns `null` for these, but the path is still rendered
(no `fill-opacity`, but the SVG `<path d=…>` is emitted). Western Sahara is the
West Africa triangle. The three `id: undefined` features (disputed territories
in the topojson) likewise pass through. **Azerbaijan being absent is a real
correctness bug** — at C1 Russian/Turkish/Persian etc. it would silently fail
to highlight.

## Decision

1. **Triangle fix — option (c): both audit and guard.** Complete the eight
   missing entries (`"732":"EH"`, `"238":"FK"`, `"260":"TF"`, `"031":"AZ"`,
   `"010":"AQ"` + a name-keyed lookup for the three `id: undefined` features:
   `"N. Cyprus":"CY-N"` is **rejected** — Northern Cyprus, Somaliland and
   Kosovo are politically disputed; we do not coin codes for them in v1.1).
   Instead: handle the five with valid M49 ids by extending the table, and let
   the three `id: undefined` features be skipped by the new guard. The
   `WorldMap` render loop adds an explicit `if (code === null) return null`
   guard so unmapped features emit zero `<path>` elements (PRD v1.1-1 wording:
   "every `<path>` element corresponds to a known region key").

   The table stays inline in `WorldMap.tsx` per the ADR 0001 addendum — still
   one consumer, no premature abstraction. We are not creating
   `src/data/m49-to-iso.ts`.

2. **Layout: side panel on the right, CSS Grid.** `App.tsx` becomes a two-cell
   grid: `grid-template-columns: 1fr 360px` with `height: 100vh` and
   `overflow: hidden` on `<main>`. Map fills the left cell; the right cell is
   a vertical flex column holding `<LanguagePicker>`, `<LanguageList>`,
   `<StatsPanel>`, `<LevelRangeSlider>` (in that order, top to bottom). Pinned
   target viewport ≥ 1280×800 — confirmed no scroll with up to 8 language
   rows in the list (panel uses `min-height: 0` + an inner flex region for the
   list so that overflow lives inside the list, not the page, if a user
   exceeds 8). Background: stays dark (`--bg`), keeps existing palette
   contrast against the white map.

3. **Picker: rewrite `LanguagePicker.tsx` in place.** Same file, same
   `onAdd(UserLanguage)` signature — App's `addLanguage` reducer is untouched
   (preserves AC 16 + new v1.1-6). Body becomes three siblings on a single
   row: `<select data-testid="language-picker-language">` listing every
   `LANGUAGES` entry sorted by `name.toLocaleLowerCase()`,
   `<select data-testid="language-picker-level">` (kept), and
   `<button data-testid="language-picker-add">` (kept). The search input and
   the option-list `<li>`s are removed.

4. **Testid contract: alias the `<select>` to `language-picker-language`,
   retire `language-picker-search` and `language-option-<id>`.** A `<select>`
   has no search box, and its `<option>`s are not the right granularity for
   `language-option-<id>` (no individual click target). `MANUAL.md` is updated:
   the search/option entries are removed and `language-picker-language` is
   added. All tests that referenced the retired ids are rewritten to drive the
   `<select>` via `selectOptions(language-picker-language, "spa")` (option
   `value` is the language id).

5. **Color palette: white country fill, blue highlight stays.** `--land`
   becomes `#ffffff`, `--accent` stays `#58a6ff`. SVG stroke becomes
   `#9ca3af` (light gray) at width `0.4` — gives country borders contrast on
   white. Highlight fill in `WorldMap` is read from `--accent` via
   `getComputedStyle` is overkill; the existing hard-coded `#3b82f6` blue is
   already sufficiently distinct from white at 0.15 (lightest opacity).
   Keep `#3b82f6` literal. App background stays dark (`--bg: #0e1116`); only
   the map's country fill is white.

## Consequences

- Two test files break and must be rewritten by QA before implementation
  starts: `tests/unit/LanguagePicker.test.tsx`, `tests/unit/App.test.tsx`,
  plus the e2e helper in `tests/e2e/v1.spec.ts` (`addLanguage` function uses
  the retired ids).
- `tests/MANUAL.md` is amended (testid contract section).
- Any component (e.g. `LanguageList`, `StatsPanel`, `LevelRangeSlider`,
  `WorldMap` body) is otherwise unchanged. Their containers move; their
  rendered output does not.
- v1 AC 1–16 continue to pass: the picker still emits the same
  `UserLanguage`, the duplicate guard still lives in `App.addLanguage`,
  persistence/coverage/stats are not touched.
- The five newly-mapped countries (EH, FK, TF, AZ, AQ) gain real
  `data-region` attrs. Azerbaijan in particular will now correctly
  highlight when a user adds a language with `"AZ"` in regions — none of
  the 10 seeded languages do, so no AC behavior changes today, but future
  dataset growth (Task 14 from ADR 0001) is unblocked.

## Follow-ups

Recorded by tech-lead during the final architectural check (v1.1 ship):

- **Task 6 white-fill technique diverged (accepted).** The ADR §6 / Task 6
  body recommended a sibling `<rect data-testid="world-map-background"
  fill="#ffffff">` as the first child of `<g>` to provide the white base.
  The implementer instead set `fill="#ffffff"` directly on uncovered
  country `<path>` elements (and `fill="#3b82f6"` on covered ones). This
  is functionally equivalent for v1.1-1, v1.1-7 and v1.1-8, produces fewer
  DOM nodes, and removes the need for a `world-map-background` testid.
  No regression; tests assert effective fill rather than a rect.
- **`--accent` is now dead code.** `:root { --accent: #58a6ff }` in
  `src/styles.css` is unused — `WorldMap.tsx` uses the literal `#3b82f6`
  per ADR §5. Left in place for now (tiny, harmless, may be picked up by
  future theming work). Cleanup is not blocking.
- **`tsconfig.node.json` / `playwright.config.ts` strict-TS errors.**
  Pre-existing scaffold issue from v1; not introduced or worsened by v1.1.
  Recorded as a follow-up for a future `chore: tighten scaffold tsconfigs`
  task. Not in scope for v1.1.
