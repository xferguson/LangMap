# v1.2 Pivot — Discovery brief

Owners: **product-designer** (Why / What stays / What gets tweaked / What gets removed / Open questions / Acceptance sketch). **tech-lead** appends "Tech-side analysis" + "Decisions for ADR 0003" below the divider.

## Why

The bare D3-rendered SVG world map shows the right shapes but doesn't feel like a map — no graticule, no continent shading, no sense of geography. v1.2 swaps the renderer to Leaflet on a low-chroma tile background so the map "reads as a map" while keeping every existing user-facing behavior. At the same time, the dataset grows from the v1.1 seed of 10 languages to a target of 200–300 (floor 100), guaranteeing that any language officially or de-facto recognized in any country is selectable. The side panel also moves from right to left for layout polish.

## What stays (preserved behavior)

- **CEFR scale and opacity ramp.** `LEVEL_OPACITY` in `src/lib/levels.ts` stays exactly as specified: `A1 0.15, A2 0.25, B1 0.40, B2 0.55, C1 0.70, C2 0.75, Native 0.75`.
- **Max-opacity dedup.** `src/lib/coverage.ts` continues to compute a region's fill opacity as `Math.max(...matchingLanguageOpacities)` (PRD AC 3).
- **Sub-national semantics.** A language whose `regions` array contains only sub-national codes (e.g. Catalan: `["ES-CT","ES-IB","ES-VC","FR-66","AD"]`) colors only those regions, never the parent country. Country-level languages (Spanish: `["ES",…]`) color the whole country. (PRD AC 4 + AC 5.)
- **Stats math.** `src/lib/stats.ts` keeps the two cards: **Communicate (B1+)** and **Get by (A1+)**, each showing distinct-country count and `(sum L1+L2) / WORLD_POPULATION`. (PRD AC 6, 7, 8.)
- **Filter independence.** The level-range filter affects map fills only; stats cards are unaffected. (PRD AC 12.)
- **Two-thumb level filter.** `src/components/LevelRangeSlider.tsx` keeps its A1 ↔ Native range and default behavior.
- **Persistence.** `localStorage` key `langmap.v1` with fault-tolerant hydration in `src/App.tsx`. (PRD AC 14, 15.)
- **Duplicate-language guard.** `App.addLanguage` semantics — re-adding an existing language updates its level rather than creating a duplicate row. (PRD AC 16, v1.1-6.)
- **`languages.ts` data shape.** `{ id, name, endonym?, speakers: { l1, l2 }, regions[] }`. New entries follow the same shape; no schema migration.
- **Single side panel housing all controls.** All four control roots (picker, list, stats, slider) remain in one panel — only the panel's position changes. (v1.1-2.)
- **Add-language row layout.** Language `<select>` + level `<select>` + Add button on one row. (v1.1-4.) See "tweaked" for the picker behavior change at scale.
- **White / blue contrast intent.** Uncovered land reads as a light, neutral base; coverage reads as a recognizable blue. The exact pixels change under Leaflet (tile background, not flat white SVG fill), but the contrast goal is preserved. (v1.1-7, v1.1-8.)
- **ⓘ tooltip on stats** noting bilingual overlap is not deduped. (PRD AC 9.)

## What gets tweaked (preferred over removal)

- **Side panel position: right → left.** Same width, same content, same internal layout — only the grid column order swaps.
- **Map renderer: D3-SVG → Leaflet GeoJSON.** Country and admin-1 polygons become Leaflet `GeoJSON` layers; their styling is driven by the same `coverageMap` produced by `src/lib/coverage.ts`. The data contract (region IDs and computed opacities) is preserved; only the rendering substrate changes.
- **Language picker UX at scale.** A flat alphabetical `<select>` with 200–300 entries is unwieldy. Open question for tech-lead — preferred direction is keeping the same single `<select>` element but layering type-ahead (browser-native search-as-you-type works on long `<select>`s) or a lightweight combobox. The contract stays: pick a language → pick a level → click Add. v1.1-6 duplicate guard must hold.
- **Dataset size: 10 → 200–300.** Same shape, same fields. Speaker counts and region lists must be sourced for every entry. The "must include if official/recognized in any country" rule is the floor.
- **Admin-1 GeoJSON subset grows.** As regional languages enter the dataset, more sub-national polygons need to be available. The subset stays curated — not the full Natural Earth admin-1 layer — but it will be larger than the ~7 entries used in v1.1.
- **Map styling.** Country borders, fill, and the absence of busy tile labels are now governed by Leaflet pane CSS and tile choice rather than direct SVG `stroke`/`fill` attributes. Visual goal: the v1.1 "white land + blue highlight" feel, but on a real map background.
- **Tests that hit `<path data-region>`.** Leaflet renders polygons via its own DOM (Canvas or SVG depending on renderer choice). Existing unit tests asserting `<path data-region="ES">` will need a different hook — e.g. attaching `data-region` via `onEachFeature` to the layer's element, or moving those assertions to e2e where Leaflet has actually mounted. Tech-lead to choose.

## What gets removed

- **Hand-rolled SVG world rendering.** The current `WorldMap.tsx` body (projection setup, manual `<path>` per feature, conditional fill via inline attributes) is replaced wholesale.
- **The `<path data-region="…">` testid contract** as v1 / v1.1 implemented it. The semantic intent ("there is a way to assert which regions are highlighted at which opacity") survives via a different DOM hook — but the literal attribute selector goes away.
- **The exact 1280×800 no-scroll AC v1.1-3.** Leaflet manages its own viewport; the constraint becomes "the page itself doesn't scroll at typical desktop sizes" without pinning to that exact pixel pair. (Tech-lead to confirm whether the strict pin is recoverable; if so, we keep it.)
- **The 10-entry seed dataset as the product target.** It stays in repo as a fallback fixture but is not the user-facing list anymore.
- **The flat-`<select>` alphabetical-only picker UX as the only mechanism.** Even if we keep `<select>`, the user experience must accommodate type-ahead at scale.

## Open product questions for tech-lead

1. Under Leaflet, can we still assert "region X has fill-opacity 0.55" at unit-test level (via attached layer DOM nodes), or do all such assertions move to e2e?
2. Sub-national + parent-country overlap with Leaflet polygons — what z-order semantics give us "Catalan-only colors CT but not ES" without polygon stacking artifacts?
3. Tile provider choice: CartoDB Positron, Stamen Toner Lite, OSM raw, or another? Preferred property: low chroma so blue overlays pop, minimal in-tile labels fighting the highlights, free for personal use, no surprise rate-limits.
4. Realistic ceiling on language count: floor 100, target 200, ceiling 300 — where do we land given sourceable speaker counts and admin-1 polygon availability?
5. The "MUST include any language that is official or recognized in any country" rule — can we invert from a country→languages list (e.g. CLDR, Wikidata, or a published JSON) to programmatically derive the must-include floor instead of hand-curating?
6. Performance at 200–300 langs × admin-1 polygons — when do map redraw or stats computation start to lag on a typical laptop? Are virtualization or memoization needed?
7. Left side-panel width — does Leaflet's map-area drive the layout differently than v1.1's `1fr 360px` grid? Stay at 360px or rethink?
8. Picker mechanism at scale — keep `<select>` (browser type-ahead), use a `<datalist>`-backed `<input>`, or add a small headless combobox component? Each affects accessibility and v1.1-6 differently.
9. What happens to languages whose `regions[]` references admin-1 codes we don't have polygons for (e.g. a regional language in a country whose admin-1 isn't in our subset)? Skip the language? Skip just those regions? Warn at build time?

## Acceptance criteria sketch (not final — these become numbered `v1.2-N` after tech-lead's pass)

- The map renders as a real-feeling map (tiles, basic geographic context) rather than a bare polygon set, with country and curated sub-national polygons highlighted at the correct CEFR opacity using the existing `coverageMap` contract.
- The side panel sits on the **left** of the viewport at desktop sizes; map fills the rest. All four control roots remain inside it.
- The language picker offers at least 100 languages (target 200–300), and includes every language that is an official or recognized language of any country in the dataset.
- All v1 ACs (1–16) and all v1.1 ACs (v1.1-1, v1.1-2, v1.1-4, v1.1-5, v1.1-6, v1.1-7, v1.1-8) continue to hold, with v1.1-3 either preserved or replaced by an equivalent "no page-level scroll at desktop" criterion to be finalized after tech-lead input.
- Sub-national languages still color only their listed regions; country-level languages still color the whole country. Overlap dedup remains max-opacity.
- Persistence, stats math, level filter independence, and duplicate-language guard are unchanged from v1.1.

## Tech-side analysis

### Answers to PD's open questions

1. **Unit-testable opacity assertions under Leaflet** — Yes, *if* we attach a stable `data-region` attribute to each layer's underlying DOM element via `onEachFeature` (calling `layer.getElement()?.setAttribute("data-region", code)` after add). With `react-leaflet`'s `<MapContainer>` mounted in jsdom (it does mount; it just can't paint), we can still `querySelector('[data-region="ES"]')` and read `getAttribute("fill-opacity")` set via the `style` callback. We keep about half of `tests/unit/WorldMap.test.tsx` and demote the v1.1-7/v1.1-8 *color* assertions to e2e (jsdom doesn't compute fill correctly through Leaflet's panes, and the blueness check needs a real renderer).
2. **Sub-national vs. parent-country z-order** — Use two separate `<GeoJSON>` layers in two distinct Leaflet panes: a `countries` pane at `zIndex: 400` and an `admin1` pane at `zIndex: 410`. `coverageMap` already returns *only* the keys that should be filled; uncovered keys get `fillOpacity: 0` (transparent). When Catalan is added, the country layer for `ES` gets `fillOpacity: 0` and the admin1 layer for `ES-CT` gets 0.55 — the admin1 polygon paints on top, so Catalonia reads as filled and the rest of Spain reads as base land. No stacking artifact because the lower pane is transparent there.
3. **Tile provider** — CartoDB Positron (`https://{s}.basemaps.cartocdn.com/light_all/{z}/{x}/{y}{r}.png`). Low chroma (greys/very pale beige), minimal labels, free for non-commercial use, served by Carto's public CDN with `{a,b,c,d}` subdomains for parallelism. Attribution required: `&copy; OpenStreetMap contributors &copy; CARTO`. Stamen Toner Lite was the runner-up but its labels fight the blue overlay more.
4. **Realistic ceiling** — 250. The constraint isn't speaker counts (Wikipedia/Ethnologue cover well past 300); it's curated admin-1 polygon availability for regional languages. Floor 100 is trivially met by official-language enumeration alone (~150 distinct official languages worldwide); 250 leaves headroom for the long tail of regional languages (Welsh, Basque, Frisian, Sami, etc.) without requiring Natural Earth's full 4,500-feature admin-1 layer.
5. **MUST-include enforcement** — Invertible. We pull a country→languages map from Wikidata SPARQL (`wdt:P37` "official language") plus the curated `unicode-org/cldr-json` `localeDisplayNames` for de-facto/recognized status. At build time, a `scripts/check-coverage.ts` script asserts that every `(country, language)` pair from the union appears either as `country in language.regions` or as a documented exception (e.g. politically disputed; see decision 6). Build fails if a must-include language is missing.
6. **Performance at 250 langs × admin-1** — Coverage recompute is O(langs × regions), which at 250 langs averaging 5 regions each is ~1,250 ops per state change — well under 1ms. The actual cost is Leaflet re-applying styles to ~400 polygons. Memoize `<GeoJSON>` per layer (the layer itself is stable across renders) and call `layer.setStyle()` imperatively in a `useEffect` keyed on `coverageMap` rather than re-mounting the GeoJSON. No virtualization needed at this scale.
7. **Side-panel width** — Stay at 360px. Leaflet auto-fills its container via `width: 100%; height: 100vh`, so the grid swap to `360px 1fr` (panel left, map right) is a one-line CSS change.
8. **Picker mechanism at 250** — Keep `<select>`. Modern browsers' built-in type-ahead is excellent on `<select>` with hundreds of `<option>`s (type "sw" → jumps to Swahili). It preserves the v1.1-6 duplicate-guard test contract verbatim (option `value` is `language.id`; `selectOption(id)` still works in RTL and Playwright). `<datalist>` would force a string-name round-trip and break id-based dedup. A custom combobox is yak-shaving.
9. **Languages with unmappable admin-1 codes** — Warn-only at build time, skip the missing region at runtime. `coverageMap` already silently no-ops on regions that don't render (it just sets a key in the map; if no polygon has that key, nothing happens). The build-time warning logs the missing `(language.id, region)` pair so we know what admin-1 features to add next. We do *not* fail the build (would block dataset growth) and do *not* skip the entire language (a language with one missing region still highlights its other regions correctly).

### Risks and unknowns

- **jsdom + Leaflet flakiness.** Leaflet expects a DOM with measurable layout (it queries `clientWidth`/`clientHeight` to position tiles). jsdom returns 0; we may need to stub `Element.prototype.getBoundingClientRect` in `tests/setup.ts` to return non-zero dimensions. This is a known react-leaflet testing gotcha.
- **Tile-load network dependency in e2e.** Playwright tests will hit Carto's CDN. We should either (a) mock `*.basemaps.cartocdn.com` to a local 1x1 PNG in CI, or (b) accept a small flake budget and add `page.waitForLoadState("networkidle")` before screenshot assertions.
- **GeoJSON file size.** A 250-language dataset paired with Natural Earth admin-1 (full) is ~5MB. We stay on the curated subset, but the countries layer still grows from ~110KB (TopoJSON) to ~600KB (GeoJSON, since Leaflet doesn't read TopoJSON natively without `topojson-client` runtime conversion). Decision: pre-convert at build time, ship GeoJSON, accept the ~500KB delta. Alternative: use `react-leaflet-topojson` — extra dep, marginal win.
- **Wikidata SPARQL drift.** The must-include enumeration script depends on Wikidata. If the SPARQL endpoint changes shape or rate-limits us, the build fails. Mitigation: cache the resolved must-include list as `src/data/must-include.json` checked into git; rerun the SPARQL query manually on dataset updates rather than on every `npm run build`.
- **Polygon-styling cost on filter drag.** The level-range slider fires onChange on every pixel of drag. With 400 polygons × imperative `setStyle()`, that's potentially 60fps × 400 = 24,000 style mutations per second on a fast drag. Mitigation: `requestAnimationFrame`-debounce the slider's onChange before it reaches `coverageMap`, or only `setStyle` polygons whose key actually changed value (diff old vs new map).

## Decisions for ADR 0003

(These will be promoted to a numbered ADR after the PRD v1.2 section is written. Listed here as draft decisions so PD can finalize ACs against them.)

1. **Leaflet renderer choice.** Use `react-leaflet` v4 with `<MapContainer>` + a memoized `<GeoJSON>` per coverage layer (one for countries, one for admin1), styled via the `style` prop reading from `coverageMap(state)`. Force the SVG renderer (`preferCanvas={false}` — the default) so individual layers expose DOM nodes for `data-region` attribute attachment via `onEachFeature`. No additional Leaflet plugins. State change reaches polygon styles imperatively: a `useEffect` keyed on the `coverageMap` reference walks the layers and calls `layer.setStyle({ fillOpacity })` rather than re-mounting the GeoJSON (which would re-create all polygons every keystroke).

2. **Tile provider.** CartoDB Positron at `https://{s}.basemaps.cartocdn.com/light_all/{z}/{x}/{y}{r}.png`, subdomains `a b c d`, max zoom 19, attribution `&copy; OpenStreetMap contributors &copy; CARTO`. Free tier; no API key. Rate-limit posture: acceptable for personal/portfolio use; documented in `tests/MANUAL.md` that bursts >10 req/s may be throttled. If we ever hit the limit we self-host via `https://github.com/CartoDB/cartodb-postgresql` or switch to MapTiler with a free key.

3. **Layout.** CSS Grid columns flip from `1fr 360px` to `360px 1fr`. Panel order in the DOM also flips: `<aside>` becomes the *first* child of `<main>`, `<WorldMap>` becomes the *second*. Panel width stays exactly 360px. `html, body, #root { overflow: hidden }` and `.app { height: 100vh }` remain. Inside `.side-panel`, vertical layout is unchanged (picker / list / stats / slider top-to-bottom).

4. **Picker mechanism at 200–300 entries.** Keep `<select data-testid="language-picker-language">` exactly as v1.1 left it. Browser-native type-ahead is sufficient at 250 entries. Existing testids and v1.1-6 (duplicate guard) are preserved with zero changes. No `<datalist>`, no custom combobox.

5. **Dataset growth strategy.** Hybrid. Source-of-truth is the union of (a) Wikidata SPARQL `SELECT ?country ?language WHERE { ?country wdt:P37 ?language }` (official languages by country, ~200 distinct languages) and (b) curated regional-language list hand-authored in `src/data/regional-languages.ts` (~50 entries: Welsh, Basque, Catalan, Frisian, Sami, Tibetan, Quechua, Guaraní, etc., chosen by us). Speaker counts come from Ethnologue with a per-entry `// source:` comment. Ceiling commitment: **250**. The "must-include if official/recognized in any country" rule is enforced by `scripts/check-coverage.ts` which runs in `pretest` and `prebuild` — fails CI if any Wikidata-derived `(country, language)` pair is missing from `LANGUAGES`.

6. **Admin-1 polygon coverage gap policy.** Warn-only. When a language references an admin-1 code we don't have a polygon for, the build script (`scripts/check-coverage.ts`) prints a `WARN: language X references region Y but no polygon exists`. The runtime simply renders nothing for that key (no error, no skipped language). The warning's job is to prompt the next dataset-growth task. We do *not* fail the build (would block adding a language whose admin-1 we'll author next sprint) and we do *not* drop the language from the picker (its other regions still color correctly).

7. **Test contract migration.** `data-region` survives via `onEachFeature: (feature, layer) => layer.on("add", () => layer.getElement()?.setAttribute("data-region", feature.properties.iso))`. Unit tests for AC 1/3/4/5/10/11 (opacity assertions) stay in `tests/unit/WorldMap.test.tsx`, gated by a jsdom `getBoundingClientRect` polyfill in `tests/setup.ts`. Tests for v1.1-7 (white fill) and v1.1-8 (blue dominance) move to `tests/e2e/v1.2.spec.ts` because they require a real renderer to compute fill. New testids: `<MapContainer>` gets `data-testid="world-map"` (replaces the old `<div>` wrapper); a hidden `<div data-testid="map-coverage-state">` next to it serializes the current `coverageMap` for fast e2e/integration snapshots without hitting tile network.

8. **Performance posture.** Memoize the two `<GeoJSON>` layers with `React.memo` keyed on dataset version (effectively constant). Recompute `coverageMap(state)` with a `useMemo` keyed on `state.languages` + `state.filter`. Apply styles imperatively on `coverageMap` change (not via re-render). RAF-debounce `LevelRangeSlider`'s onChange to coalesce drag events. Only `setStyle` polygons whose entry in the new map differs from the old map (diff-based update). No virtualization. No GeoJSON precomputation beyond the build-time TopoJSON→GeoJSON conversion script in `scripts/build-geo.ts`.

9. **Status of v1.1-3 (no scroll at 1280×800).** Replaced with a softer equivalent: "no page-level scroll at desktop viewports ≥ 1024×768". Leaflet's container fills available height/width and never overflows; the side panel's internal list is the only scrollable region (already enforced by `min-height: 0` + inner `overflow-y: auto`).

10. **Backwards-compat boundary.** Confirmed unchanged: `langmap.v1` localStorage key, payload schema (`{ languages: UserLanguage[], filter: { min: Level, max: Level } }`), `coverageMap` return type (`Map<RegionRef, number>`), `LANGUAGES` element shape (`{ id, name, endonym?, speakers: { l1, l2 }, regions[] }`), all four panel components' public props. A v1.1 user reloading after deploying v1.2 keeps their saved languages intact.
